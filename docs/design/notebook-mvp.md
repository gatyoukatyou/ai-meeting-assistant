# 設計: 自動筆記手帳MVP（#160）— 2プロファイル・共通保存基盤

> **状態**: 設計ドキュメント（コード変更なし）。handoff/status-2026-07-15.md の判断事項1・2と、
> 「会議モードと手帳機能の併存＝2プロファイル・共通保存層」方針のHUMAN承認を受けて作成。
> 実装はここに定義するPR分割に従い、1PRずつレビュー・mergeして進める。

## 0. 製品定義（承認済み）

**短い業務会話をすぐ記録し、AIで構造化して保存し、後から探して別ツールへ渡せるアプリ。**

- 既存の「会議にAIが参加する」コンセプトは**置き換えない**。共通パイプライン
  （録音 → 文字起こし → AI加工 → 保存）の上の1プロファイルとして再配置する。
- アプリ内AIは小さく保つ。Todoist連携・詳細分析・会話履歴横断のAI対話は**内蔵しない**。
  front matter付きMarkdown出力を介して外部エージェントへ渡す。
- AIが抽出したアクションは常に「候補」として提示し、HUMAN確認後に外部へ渡す。

## 2. アーキテクチャ: 2プロファイル・共通パイプライン

```
            ┌─ 会議プロファイル（既存） ──────────────┐
起動 ─┬─▶  │ 会議情報設定 → 録音 → リアルタイムAI     │─┐
      │    │ （要約/意見/アイデア/相談）→ 停止 → 議事録 │ │
      │    └───────────────────────────┘ │
      │                                                  ▼
      │    ┌─ メモプロファイル（新規） ─────────┐   共通後処理
      └─▶  │ ワンタップ録音 → 文字起こしのみ → 停止 │─▶ AI構造化(4項目+メタ提案)
           └─────────────────────────┘   → 確認・修正 → 保存
                                                          ▼
                                     共通保存層（蓄積型レコード）→ 検索・閲覧 → MD出力
```

- **プロファイルの決め方**: 起動後のワンタップ録音は常にメモプロファイル。
  リアルタイムAIが必要なときだけ会議プロファイルを明示的に選ぶ。モードは2つで固定し、増やさない。
- **中間ケースの扱い**: 後処理（構造化・保存・検索・出力）は共通なので、
  どちらのプロファイルで録っても最終的な記録はほぼ同じになる。誤選択の損失は小さい。
- 会議プロファイルの録音停止後にも同じ構造化・保存を通す。議事録（`minutes`）は
  レコードの1フィールドとして共存させる（既存機能の後退なし）。

## 3. データ構造（判断事項2の具体化）

### 3.1 保存レコード（IndexedDB、スキーマv3）

既存フィールド（`buildHistoryRecord()` が生成する `id / title / createdAt / updatedAt /
transcript / durationSec / summaryPreview / exportMarkdown` ほか）を維持し、以下を追加する。

```js
{
  // --- 既存（維持） ---
  id, title, createdAt, updatedAt, transcript, durationSec, summaryPreview,

  // --- v3で追加 ---
  schemaVersion: 3,
  profile: 'meeting' | 'memo',
  category: '会議・打合せ' | '相談・確認' | '指示・依頼' | 'アイデア' | 'その他',
  tags: ['資金繰り', ...],          // AI提案 + 手動修正
  status: 'raw' | 'organized',      // 未整理 / 整理済み
  structured: {
    keyPoints: [...],               // 要点
    decisions: [...],               // 決定事項
    actionCandidates: [...],        // アクション候補（タスクではない）
    openQuestions: [...]            // 未解決事項
  },
  minutes: '...'                    // 会議プロファイルの議事録（既存機能、任意）
}
```

- カテゴリは上記5種で開始し、当面増やさない（分類が面倒になると記録されなくなる）。
- 元の文字起こし（`transcript`）は必ず保存する。構造化は追加であり、元データを置き換えない。

### 3.2 保存層の変更方針

- `MAX_RECORDS = 5` の自動削除を撤廃する。削除は明示操作のみ。
- 上限は件数ではなく容量で管理する: `navigator.storage.estimate()` を監視し、
  逼迫時に警告＋古い記録のMD一括出力を促す。目安として1時間会議の文字起こしは
  数百KB程度であり、通常利用で数年分は保存可能。
- v2→v3マイグレーション: 既存レコードに `schemaVersion:3, profile:'meeting',
  category:'会議・打合せ', tags:[], status:'raw'` を付与する（データ損失なし・可逆）。
- `navigator.storage.persist()` を要求し、ブラウザによる自動削除を抑止する（拒否時は設定画面に注意表示）。

### 3.3 Markdown出力（外部エージェント連携の契約）

1件単位の出力形式。**このfront matterが外部エージェントとのインターフェース契約**であり、
フィールドの削除・改名はHUMAN承認事項とする。

```markdown
---
id: conversation-20260715-001
created_at: 2026-07-15T13:30:00+09:00
profile: memo
category: 相談・確認
tags:
  - 資金繰り
status: organized
duration_sec: 312
---
# ○○案件についての確認

## 要点
- ...

## 決定事項
- ...

## アクション候補
- [ ] ...   ← 候補であり確定タスクではない（見出しにもその旨を明記）

## 未解決事項
- ...

## 議事録            ← 会議プロファイルかつ生成済みの場合のみ

## 文字起こし
...（全文）
```

- 既存 `export-service.generateMarkdown()` を拡張し、front matterと4項目セクションを追加する。
  既存のエクスポート項目選択（議事録・要約・コスト等）は維持。
- 一括出力: 選択した複数レコードを個別ファイルとしてzip、または連結1ファイル（MVPでは連結で開始）。

## 4. 画面遷移

### 4.1 メモプロファイル（新規の主フロー）

```
起動 → [● 録音]ワンタップ（設定不要、タイトル・カテゴリは後付け）
  → 録音中（状態バッジ: 録音中/中断/処理中を明示 — #159小改良と共通）
  → [■ 停止]
  → 停止直後画面: 文字起こし全文 + [✨ 整理する] ボタン（目立たせる）
  → AI構造化（1回のLLM呼び出しで 4項目 + タイトル案 + カテゴリ案 + タグ案 を生成）
  → 確認・修正画面:
      - タイトル/カテゴリ/タグ: AI提案値をプリセット、タップで修正
      - 4項目: インライン編集可
      - 補助の自由入力欄1つ（「この会話で私がやるべきことは？」等の追加指示 → 再生成）
  → [保存] → status: organized で保存 → 履歴一覧へ
  ※「整理する」を押さずに閉じた場合も transcript は status: raw で自動保存（記録を失わない）
```

### 4.2 会議プロファイル（既存 + 合流点）

既存フロー（会議情報設定 → 録音 → リアルタイムAI → 停止 → 議事録生成）は変更しない。
停止後の保存時に §3.1 のレコード形式で保存され、`category` 初期値は「会議・打合せ」。
必要なら同じ確認・修正画面でメタデータを編集できる。

### 4.3 履歴（手帳の本体）

- 日付降順の一覧（現状の「直近5件モーダル」を置き換え）。各行: タイトル・日付・カテゴリ・タグ・status。
- 絞り込み: カテゴリ（5種チップ）、タグ、期間。
- 全文検索: `transcript` と `structured` を対象。MVPでは全件走査の単純フィルタで開始し、
  性能問題が実測されてから索引化を検討する（早すぎる最適化をしない）。
- 詳細画面: 構造化4項目 → 議事録（あれば）→ 文字起こし の順に表示。[MD出力] [削除（確認付き）]。

## 5. スコープ外（MVPでやらないこと）

- Todoist等への自動登録（MD出力を外部エージェントに渡す）
- 会話履歴を横断したAI対話・分析
- スキン刷新（#159は§4.1の状態明示・導線改善のみに限定）
- クラウド同期・複数端末共有
- 検索の索引化（実測で必要になるまで）

## 6. セキュリティ・情報管理（MVPでの対応と運用注意）

| 論点 | MVPでの対応 |
|---|---|
| データの置き場所 | 端末内IndexedDBのみ。サーバー保存なし（現行と同じ） |
| APIへ送る範囲 | 構造化時に文字起こしをLLMへ送信（既存の要約・議事録と同じ経路・同じ鍵管理）。それ以外を送らない |
| アクション誤抽出 | 常に「候補」表記。アプリから外部への自動登録をしない |
| 端末紛失 | 保存データは端末のOS保護に依存（現行と同じ）。README/PRIVACYに明記 |
| エクスポートMD | 出力後の管理は利用者責任である旨を出力UIに一言表示 |
| 録音の同意 | アプリでは強制できない。運用ルール（録音を相手に伝える）をREADMEに明記 |

機微情報（人事・取引先・資金等）のカテゴリ別の保存制限はMVPでは設けない（運用で対応）。

## 8. MVP受入条件（完了の定義）

1. ワンタップで録音開始し、停止→整理→保存が3タップ以内で完了する
2. 保存件数が5件を超えても過去の記録が消えない（v2からの移行データ含む）
3. 履歴を日付・カテゴリ・タグ・全文検索で絞り込める
4. 1件をfront matter付きMDで出力でき、外部エージェントがメタデータを機械的に読める
5. 既存の会議フロー（リアルタイムAI・議事録・エクスポート）が後退しない
6. 録音回帰テスト11シナリオがALL PASSのまま
